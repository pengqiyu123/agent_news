# 公众号内容表现复盘

这份文档保存真实发表记录分析得到的观察性运营结论，供后续 Agent 选题、写标题、写正文和复盘时读取。它不是固定工作流，也不是发布成功证明；数据仍以 `wechat.analyze_publish_metrics` 和微信后台发表记录为准。

## 解释边界

- 这里的结论只适用于当前公众号的科技内容观察样本，不代表全平台规律。
- 当前数据不是标题 A/B 测试，没有随机分流，也没有控制发布时间、封面、群发状态和选题子赛道。
- 任何 Agent 都不得输出“标题导致阅读高”这类因果判断，只能说“历史样本中更贴近高表现结构”。
- 群发通知文章、未开启群发通知发表文章、刚发布未满 24 小时文章必须分层看；不能直接横比阅读量。
- `content_strategy_profile.evidence_level` 为 `weak` 或 `medium` 时，只能作为提示，不得触发硬规则、自动否稿或自动加权。

## 观察性运营结论

1. 阅读高度头部集中。前 4 篇贡献 446 阅读，约占总阅读的 93.3%。这说明当前账号历史样本存在头部效应，但不能单独归因到标题。
2. 当前表现较好的样本多把 AI、芯片、硬件变化落到钱包、手机、电脑、电力、企业成本、供应链等现实后果。它是选题和标题假设，不是因果规律。
3. 纯“科技速览”“今日短讯”类标题在当前样本中偏弱，除非 5 条新闻之间有共同主线，例如涨价、缺电、裁员、监管、成本外溢。
4. 当前点赞、留言、推荐、划线、赞赏、转载全为 0，暂时不要用这些指标判断内容质量。现阶段阅读和分享只是“可观察度最高”的指标，不等于完整后台分析。
5. 发表记录抽取还存在噪声，例如 `开启留言` 这类非文章入口会混入统计。分析结论需要优先看正常文章标题和公开文章 URL。
6. 最近一篇 `今日5条科技短讯｜白宫叫停OpenAI新模型、Adobe收购Topaz、游戏训练AI获23亿押注` 当前为 0 阅读，先不要急判死刑；但它强化了一个提醒：短讯合集不能只堆信息点，必须提炼共同主线。

## 写稿规则

后续 Agent 写公众号稿时优先使用以下策略，但只能作为当前账号内的观察性偏好：

- 标题前半句放具体公司、产品、价格、事故、裁员、涨价、成本、电力、芯片等硬信息。
- 标题后半句给现实后果：`账单转给消费者`、`电网开始吃紧`、`手机可能更贵`、`开发者成本下降`、`企业开始重新排队`。
- 对 5 条短讯合集，必须先提炼共同主线。不要只写“5条短讯/科技速览”，要写成“5条新闻共同说明了什么变化”。
- 正文开头先讲读者为什么该关心，再讲事件。少用行业内自嗨词，多写“这会影响谁、怎么影响、时间点是什么”。
- 如果同标题出现多篇，复盘和删除都必须传 `url` 精确定位，不能只靠标题。
- 小于 24 小时的新文章只看趋势，不做最终质量结论。
- 如果群发状态未知，复盘报告必须把“群发通知状态未分层”列为混杂因素。

## 建议标题结构

优先：

- `A翻车、B涨价、C缺电：AI正在把成本推向普通人`
- `某公司做了某事：某个行业账单开始重算`
- `某硬件/模型/政策变化：谁会先受影响`
- `5条短讯｜A、B、C、D、E背后，是同一个成本/监管/供应链变化`

谨慎：

- `今日科技速览`
- `某某发布新产品`
- `某某宣布合作`
- 只有技术名词、没有现实后果的标题

## 下次复盘入口

常规复盘：

```powershell
.\.venv\Scripts\python.exe -m agent_news run wechat.analyze_publish_metrics limit=15 max_pages=2
```

单篇复盘优先传 URL：

```powershell
.\.venv\Scripts\python.exe -m agent_news run wechat.review_content_performance url="https://mp.weixin.qq.com/s/..." max_pages=2 lookback_runs=5
```

如果运行中的常驻服务提示 `Operation 'wechat.review_content_performance' not registered`，说明服务进程还是旧版本。先执行 `stop.bat`，再用项目虚拟环境重新运行 CLI。
